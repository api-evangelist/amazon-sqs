<p>This post is part 3 of the three-part series ‘Enabling high availability of Amazon EC2 instances on&nbsp;<a href="https://aws.amazon.com/outposts/servers/" rel="noopener noreferrer" target="_blank">AWS Outposts</a>&nbsp;servers’. We provide you with code samples and considerations for implementing custom logic to automate&nbsp;<a href="https://aws.amazon.com/ec2/" rel="noopener noreferrer" target="_blank">Amazon Elastic Compute Cloud (EC2</a>) relaunch on Outposts servers. This post focuses on guidance for using Outposts servers with third party storage for boot and data volumes, whereas <a href="https://aws.amazon.com/blogs/compute/enabling-high-availability-of-amazon-ec2-instances-on-aws-outposts-servers-part-1/" rel="noopener noreferrer" target="_blank">part 1</a> and&nbsp;<a href="https://aws.amazon.com/blogs/compute/enabling-high-availability-of-amazon-ec2-instances-on-aws-outposts-servers-part-2/" rel="noopener noreferrer" target="_blank">part 2</a> focus on automating EC2 relaunch between standalone servers. Outposts servers support integration with&nbsp;<a href="https://www.dell.com/en-us/shop/storage-servers-and-networking-for-business/sf/power-store">Dell PowerStore</a>,&nbsp;<a href="https://www.hpe.com/us/en/storage/alletra.html">HPE Alletra Storage MP B10000&nbsp;systems</a>, <a href="https://www.netapp.com/data-management/ontap-data-management-software/">NetApp on-premises enterprise storage arrays</a>, and <a href="https://www.purestorage.com/products/nvme/flasharray-x.html">Pure Storage FlashArray</a>.</p> 
<p>Outposts servers provide compute and networking services that are designed for low-latency, local data processing needs for on-premises locations such as retail stores, branch offices, healthcare provider locations, or environments that are space-constrained. Outposts servers use <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/InstanceStorage.html">EC2 instance store storage</a> to provide non-durable block-level storage to the instances running stateless workloads. For applications that require persistent storage, you can create a three-tier architecture by connecting your Outposts servers to a third-party storage appliance. In this post, you will learn how to implement custom logic to provide high availability (HA) for your applications running on Outposts servers using two or more servers for N+1 fault tolerance. The code provided is meant to help you get started, and can be modified further for your unique workload needs.</p> 
<h2>Overview</h2> 
<p>In the following sections we will show how custom logic can be used to automate EC2 instance relaunch between two or more Outposts servers using boot and data volumes on third party storage. If your EC2 instance fails while using this solution, an <a href="https://aws.amazon.com/cloudwatch/" rel="noopener noreferrer" target="_blank">Amazon CloudWatch</a> alarm monitoring the EC2 StatusCheckFailed_Instance metric of your source EC2 instance will be triggered, and you will receive an <a href="https://aws.amazon.com/pm/sns/?trk=a074e8bd-fe9a-4ee3-ad49-f731a39ed149&amp;sc_channel=ps&amp;ef_id=Cj0KCQjw0NPGBhCDARIsAGAzpp09acksHrmkVGsgrQOD0PemL3_g9NKKPFW-WSwnyrwz3JofgE8cE-gaAquyEALw_wcB:G:s&amp;s_kwcid=AL!4422!3!658520967038!!!g!!!19852662602!149878732060&amp;gad_campaignid=19852662602&amp;gbraid=0AAAAADjHtp9ku4mrGWr4lYItA40Hw968W&amp;gclid=Cj0KCQjw0NPGBhCDARIsAGAzpp09acksHrmkVGsgrQOD0PemL3_g9NKKPFW-WSwnyrwz3JofgE8cE-gaAquyEALw_wcB" rel="noopener noreferrer" target="_blank">Amazon Simple Notification Service</a> (Amazon SNS) notification. An <a href="https://aws.amazon.com/pm/lambda/?trk=a968e0d4-b96f-4cef-9ed9-be59b3588c76&amp;sc_channel=ps&amp;ef_id=Cj0KCQjw0NPGBhCDARIsAGAzpp0GWTfgKmF6tf6S4dDuyzy-xKlzC-ovRXnP2NkmRMM5JtWj8a87UuQaAgGvEALw_wcB:G:s&amp;s_kwcid=AL!4422!3!652240143523!e!!g!!amazon%20lambda!19878797032!147151597893&amp;gad_campaignid=19878797032&amp;gbraid=0AAAAADjHtp87KK8zRjKPBySDn4-2cQ836&amp;gclid=Cj0KCQjw0NPGBhCDARIsAGAzpp0GWTfgKmF6tf6S4dDuyzy-xKlzC-ovRXnP2NkmRMM5JtWj8a87UuQaAgGvEALw_wcB" rel="noopener noreferrer" target="_blank">AWS Lambda</a> function will then relaunch your EC2 instance onto the destination Outposts server that you’ve set up for resiliency. This is done using a launch template created during setup, and the script will connect your relaunched instance to the existing boot and data volumes on your third party storage appliance. This storage device provides shared storage for your Outposts servers. If a single server fails, new instances can connect to existing volumes on the array. This allows for a zero data loss <a href="https://docs.aws.amazon.com/whitepapers/latest/disaster-recovery-of-on-premises-applications-to-aws/recovery-objectives.html" rel="noopener noreferrer" target="_blank">Recovery Point Objective (RPO)</a> and a <a href="https://docs.aws.amazon.com/whitepapers/latest/disaster-recovery-of-on-premises-applications-to-aws/recovery-objectives.html" rel="noopener noreferrer" target="_blank">Recovery Time Objective (RTO)</a> equaling the time it takes to launch your EC2 instance. Take advantage of the features on your storage appliance for configuring data durability and resiliency to hardware failures, and make sure that you are regularly backing up your SAN volumes.</p>

<h3>Prerequisites</h3> 
<p>The following prerequisites are required to complete the walkthrough:</p> 
<ul> 
 <li>Two Outposts servers that can be set up as an&nbsp;<a href="https://aws.amazon.com/blogs/architecture/disaster-recovery-dr-architecture-on-aws-part-i-strategies-for-recovery-in-the-cloud/" rel="noopener noreferrer" target="_blank">active-active or active-passive</a> resilient pair.</li> 
 <li>For workloads with a low threshold for downtime, ensure that your secondary Outpost server that’s used for recovery has a unique service link connection.</li> 
 <li>Outposts servers must be colocated within the same Layer 2 (L2) network.</li> 
 <li>Network latency between the Outposts servers must not exceed 5ms round trip time (RTT).</li> 
 <li>A storage appliance that supports the iSCSI protocol. Credentials to manage the storage appliance initiator/target mappings. <a href="https://aws.amazon.com/blogs/compute/new-simplifying-the-use-of-third-party-block-storage-with-aws-outposts/" rel="noopener noreferrer" target="_blank">See Simplifying the use of third-party block storage with AWS Outposts</a> for more information.</li> 
 <li>If you’re setting this up from an&nbsp;<a href="https://docs.aws.amazon.com/outposts/latest/userguide/sharing-outposts.html" rel="noopener noreferrer" target="_blank">Outposts consumer account</a>, you must configure <a href="https://aws.amazon.com/blogs/mt/monitoring-best-practices-for-aws-outposts/" rel="noopener noreferrer" target="_blank">Amazon CloudWatch cross-account observability</a>&nbsp;between the consumer account and the Outposts owning account to view Outposts metrics in your consumer account.</li> 
 <li>Create <a href="https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/ec2-launch-templates.html">launch templates</a> for the EC2 instances that you want to protect, the launch wizard will help you create these.</li> 
 <li>Credentials with permissions for <a href="https://www.google.com/url?sa=t&amp;source=web&amp;rct=j&amp;opi=89978449&amp;url=https://aws.amazon.com/cloudformation/&amp;ved=2ahUKEwjZmOfljKGQAxWIFFkFHXEGFS4QFnoECB0QAQ&amp;usg=AOvVaw2O20tPzwYsGu9e_oSCbvzG" rel="noopener noreferrer" target="_blank">AWS CloudFormation</a>, <a href="https://www.google.com/url?sa=t&amp;source=web&amp;rct=j&amp;opi=89978449&amp;url=https://aws.amazon.com/ec2/&amp;ved=2ahUKEwjq3rnRjKGQAxW6L1kFHbu9NZgQFnoECBkQAQ&amp;usg=AOvVaw3MI5OycyIjdz9NSdetTohX" rel="noopener noreferrer" target="_blank">Amazon EC2</a>, and (optional) <a href="https://aws.amazon.com/secrets-manager/" rel="noopener noreferrer" target="_blank">AWS Secrets Manager</a> if authentication is required. IAM Permission Examples.md is provided in the repository.</li> 
 <li>A Windows or Linux host that can access the storage appliance and your AWS account (management computer).</li> 
 <li><a href="https://us-east-1.console.aws.amazon.com/marketplace/search/listing/prodview-ytzcqvandumqm" rel="noopener noreferrer" target="_blank">AWS Outposts iPXE Amazon Machine Image</a> (AMI) from the <a href="https://www.google.com/url?sa=t&amp;source=web&amp;rct=j&amp;opi=89978449&amp;url=https://aws.amazon.com/marketplace&amp;ved=2ahUKEwig5aGHmaGQAxVQwskDHUdYHS4QFnoECBIQAQ&amp;usg=AOvVaw2kR1wc3JVnglAce4z8i-IH" rel="noopener noreferrer" target="_blank">AWS Marketplace</a>.</li> 
 <li><a href="https://www.python.org/" rel="noopener noreferrer" target="_blank">Python</a>&nbsp;3.8 or later (recommended) is used to run the&nbsp;init.py&nbsp;script that dynamically creates a&nbsp;CloudFormation&nbsp;stack in the account specified as an input parameter.</li> 
 <li><a href="https://aws.amazon.com/sdk-for-python/" rel="noopener noreferrer" target="_blank">AWS SDK for Python (Boto3)</a> version 1.26.0 or later recommended.</li>

<h2>Walkthrough</h2> 
<p>The first step is to deploy an EC2 instance configured to boot from a volume on the third-party storage that is prepared with an OS boot image. This step uses the launch wizard portion of the solution.</p> 
<ol> 
 <li>Download and extract the OutpostServer_Recovery_3Pstorage repository to the management computer that has the AWS SDK for Python (Boto3) and Python installed.</li> 
 <li>Run launch_wizard from the sample-outposts-third-party-storage-integration directory. You can run interactively or provide arguments for region, subnet, iPXE AMI, storage vendor, storage management ip, and credentials.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-2.png"><img alt="" class="alignnone wp-image-25766 size-full" height="740" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-2.png" width="1428" /></a></p> 
<p>Figure 2 – Running launch wizard</p> 
<ol start="3"> 
 <li>When prompted for a feature name, enter sanboot.</li> 
 <li>For Guest OS type, enter in Linux or Windows.</li> 
 <li>When prompted “Do you want to continue with this unverified AMI?”, select Y.</li> 
 <li>The launch wizard will provide a list of instance types available on the Outpost server associated with the subnet you specified. Enter the instance type that you want to use.</li> 
 <li>The launch wizard will now prompt you for optional EC2 Key Pair, Security Group, and Instance Profile settings for the EC2 instance that you are launching.</li> 
 <li>Next, the launch wizard prompts you to specify an instance name. Note that specifying an instance name is required to set up automated instance recovery because the instance name is used as part of the recovery process.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-3.png"><img alt="" class="alignnone wp-image-25767 size-full" height="565" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-3.png" width="1432" /></a></p> 
<p>Figure 3 – Taking user input for variable values</p> 
<ol start="9"> 
 <li>The launch wizard prompts for root volume size. This is the root volume that the iPXE AMI boots from. The default is a 1GB volume on the Outpost server instance storage.</li> 
 <li>Next, the launch wizard prompts you to select which third party storage controller you want to use based on the management ip that you specified. In this example, we are using NetApp, so I select a NetApp Storage Virtual Machine (SVM) named outpost_iscsi.</li> 
 <li>If the connection to the storage array is successful and the protocol is available (iSCSI or NVMe over TCP) you are provided additional storage options for initiator group and logical unit number (LUN).</li> 
 <li>In this example, we are using NetApp with iSCSI, so I can select an existing initiator group or create a new one.</li> 
 <li>You can specify an existing initiator qualified name (IQN), or the launch wizard can generate a new one. <strong>IMPORTANT:</strong> Make sure that IQNs are unique to each instance because duplicates can cause data corruption.</li> 
 <li>Next the launch wizard prompts which LUN’s you want to connect to this instance. For this example, I am going to use a Windows Server 2022 boot volume that I already created on the NetApp storage array.</li> 
 <li>You are now asked which storage array target interface you want to use for connecting to these LUNs.</li> 
 <li>The launch wizard provides the capability to specify guest OS scripts to customize the OS after sanboot. Combining this capability with storage array cloning provides a streamlined process for deploying new instances.</li> 
 <li>The launch wizard now displays the EC2 user data template that it generated for use with the iPXE AMI and asks if you want to proceed with launching the instance.</li> 
 <li>After the EC2 instance is launched, select yes to proceed with automated instance recovery setup.</li> 
</ol>

<h3>Generating EC2 launch templates for recovery and failback</h3> 
<p>In the second step, we are generating EC2 launch templates for the EC2 instance launched in step 1. Launch templates can be generated for the primary and secondary Outpost servers. The launch template for the secondary Outpost server can be used for automated or manual recovery of the EC2 instance. Failback to the primary Outpost server is manual using the primary launch template.</p> 
<ol> 
 <li>Select the instance that you want automated recovery for and select the subnet that you launched the instance in. This subnet represents the primary Outpost server that the instance is running on.</li> 
</ol> 
<p><a href="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-5.png"><img alt="" class="alignnone wp-image-25769 size-full" height="809" src="https://d2908q01vomqb2.cloudfront.net/1b6453892473a467d07372d45eb05abc2031647a/2026/02/25/ComputeBlog-2445-image-5.png" width="891" /></a></p> 
<p>Figure 5 – Selecting subnets for EC2 instance relaunch</p> 
<ol start="2"> 
 <li>When prompted to create a second launch template for Outpost server recovery, select yes, and then select to use the same instance (for recovery on different Outpost server).</li> 
 <li>When you get a list of available subnets, select the subnet that’s associated with your secondary Outpost server. This is the server that the EC2 instance will be launched on in the event of the EC2 StatusCheckFailed_Instance metric triggers the CloudWatch alarm.</li> 
 <li>You will see both launch templates created successfully.</li> 
</ol> 
<h3>Deploying automated EC2 instance recovery</h3> 
<p>The third step creates a CloudFormation template for monitoring, notifications, and automated recovery of the EC2 instance deployed in step 1. The CloudFormation template automatically captures the instance and secondary launch template information necessary for automatic recovery.</p> 
<ol> 
 <li>Select Y to set up automated recovery. This will create a CloudFormation stack.</li> 
 <li>Provide a name and description for the CloudFormation stack.</li> 
 <li>Select whether you want automated recovery or notification only. This provides flexibility to choose manual or automatic recovery based on whether you want to verify the primary Outpost server is down before initiating recovery.</li> 
 <li>In the AWS CloudFormation console, monitor the CloudFormation stack creation process.</li> 
</ol>

<ol start="5"> 
 <li>After the CloudFormation Stack is complete, you have successfully deployed an EC2 instance using third party storage for boot and data volumes on a primary Outpost server. You also created instance recovery capabilities by using the Amazon Outpost server automated recovery solution for third party storage.</li> 
 <li>You can verify whether the EC2 StatusCheckFailed_Instance is healthy under the Alarms section in the Amazon CloudWatch console.</li> 
</ol> 
<h2>Considerations</h2> 
<p>The logic discussed in this post relies on the secondary destination Outposts server having a connected service link. For more information about how to create a highly available service link connection for your Outpost servers, see the <a href="https://docs.aws.amazon.com/whitepapers/latest/aws-outposts-high-availability-design/anchor-connectivity.html" rel="noopener noreferrer" target="_blank">Networking section</a> of AWS Outposts High Availability Design and Architecture Considerations whitepaper.</p> 
<h2>Clean up</h2> 
<p>Confirm whether it is safe to terminate the Amazon EC2 instance that you launched with this walkthrough. The operating system and data volumes are on the third party storage, so EC2 instance termination only removes the iPXE AMI from the Outposts server instance storage. To clean up, complete the following steps.</p> 
<ol> 
 <li>Terminate the Amazon EC2 instance. Then, verify that the Instance state is <strong>Terminated</strong> to ensure that the instance is not using Outposts server resources.</li> 
 <li>Delete the Amazon EC2 Launch Templates associated with the Amazon EC2 instance that you terminated. The names of the launch templates that were automatically generated will start with ‘lt-‘, followed by the instance name and the instance id. If you generated a recovery launch template, it will have a ‘-recovery’ suffix in the name.</li> 
 <li>Delete the AWS CloudFormation Stack. The Stack name will start with ‘autorestart-‘ followed by the Amazon EC2 instance name.</li> 
 <li>Clean up your initiators, initiator group, and LUNs on the third party storage array.</li> 
</ol> 
<h2>Conclusion</h2> 
<p>With the use of custom logic through AWS tools such as CloudFormation,&nbsp;CloudWatch, Amazon SNS, and&nbsp;AWS Lambda, you can architect for HA for stateful workloads on Outposts server. By implementing the custom logic in this post, you can automatically relaunch EC2 instances running on a source Outposts server to a secondary destination Outposts server if an instance fails, and connect to existing volumes on a shared storage appliance for recovery. This also reduces the downtime of your applications in the event of a hardware or service link failure. The code provided in this post can be further expanded upon to meet the unique needs of your workload.</p>
